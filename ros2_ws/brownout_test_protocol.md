# 브라운아웃(전압강하) 원인 확정 실험 절차

> 가설: **모터 드라이버 5V를 Pi에서 공급 → 모터 起動 인러시 전류로 Pi 5V 레일이 순간 sag → under-voltage → USB(카메라) 리셋 → "합치면 끊김"**
>
> 도구: `brownout_detect.py` (sticky bit 고속 폴링) — ms 단위 순간 강하도 래치로 포착.
> 핵심 명령: `vcgencmd get_throttled` 의 비트
> - `0x1` 현재 under-voltage / `0x10000` under-voltage **발생 이력**(래치)

---

## 0. 준비
```bash
vcgencmd get_throttled        # 현재 값 확인 (지금은 0x50000 = 과거 강하 이력 있음)
```
깨끗한 판정을 위해 **리부트로 플래그 초기화** 후 시작 권장:
```bash
sudo reboot
# 재부팅 후
vcgencmd get_throttled        # 0x0 이어야 함
```

---

## 1. 베이스라인 — 모터 빼고 전부 구동 (Pi+카메라 무죄 확인)
한 터미널:
```bash
python3 ~/ros2_ws/brownout_detect.py
```
다른 터미널들에서 **모터만 빼고** 카메라2 + aruco + gesture + state_machine 구동.
→ 몇 분간 새 under-voltage 비트가 **안 박히면**: Pi + 카메라 + CPU 부하 자체는 문제 아님.

---

## 2. 트리거 — 모터 起동 (가설 검증)
1번 상태 그대로 두고, 모터를 실제로 起동시킨다.
- 起동 **바로 그 순간** `brownout_detect.py` 에 빨간 `⚡UNDER-VOLTAGE` 줄이 뜨고
  카메라가 끊기면 → **가설 1차 확정** (起동 시각 ↔ 비트 박힌 시각 일치).

---

## 3. 결정적 반증 테스트 — 전원 분리 (인과 100% 확정)
> 별도 전원 있음 → 이 단계로 확정 가능.

배선 변경:
- 모터 드라이버 5V(모터/로직 전원)를 **Pi가 아닌 별도 5V 소스(배터리/별도 어댑터)** 에서 공급.
- **GND는 Pi와 반드시 공통(공통접지)** 으로 묶을 것. (안 묶으면 신호 기준 어긋남)
- PWM/제어 신호선은 그대로 Pi GPIO 유지.

같은 절차(1→2) 반복:
- 모터 起동해도 under-voltage 비트 **안 박히고** 카메라 **안 끊기면**
  → **원인 = 공유 5V 전원, 확정.**
- 그래도 끊기면 → 전원 외 다른 원인(USB 대역폭/접지/back-EMF 등) 추가 조사.

## 판정 요약
| 1번(모터X) | 2번(모터O, 공유) | 3번(모터O, 분리) | 결론 |
|---|---|---|---|
| 깨끗 | UNDER-VOLT 발생 | 깨끗 | **공유전원 브라운아웃 확정** ✅ |
| 깨끗 | UNDER-VOLT 발생 | 여전히 발생 | 전원+@ 다른 요인 |
| 깨끗 | 깨끗 | — | 전원 아님, 다른 원인 조사 |

## 확정 시 해결 방향 (참고)
- 모터 전원을 Pi에서 분리(별도 5V/배터리, 공통접지) — 위 3번이 곧 해결책.
- 모터 드라이버 입력단에 **대용량 전해 커패시터**로 인러시 완충.
- Pi 전원은 정격 5V/5A(Pi5 공식 27W PD) 어댑터 사용.
